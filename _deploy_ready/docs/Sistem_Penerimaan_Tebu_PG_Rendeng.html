Detail Setiap Langkah

Langkah 1-2: Kedatangan & Verifikasi Lokasi + Pos Binding

Saat petugas membuka aplikasi dan memulai pencatatan baru, sistem secara otomatis menjalankan validasi lokasi multi-layer. Sistem mengecek apakah petugas berada dalam radius 200 meter dari koordinat pos pantau yang sudah didaftarkan. Selain itu, sistem memverifikasi bahwa pos_id yang terdeteksi dari GPS cocok dengan pos_id yang ditugaskan ke petugas tersebut di database. Jika terdeteksi fake GPS, lokasi tidak valid, atau petugas berada di pos yang bukan penugasannya, petugas tidak bisa melanjutkan input data.

Langkah 3: Input Nomor Kontrak

Petugas mengetikkan nomor kontrak. Sistem melakukan lookup ke database dan secara otomatis menampilkan nama petani, alamat, dan nomor WhatsApp. Jika kontrak tidak ditemukan, petugas dapat memilih mode "Input Manual" untuk mengisi data secara langsung, tetapi data tersebut akan ditandai sebagai "Perlu Verifikasi" di sistem pusat.

Langkah 4: Foto Truk & OCR Pelat Nomor

Petugas mengambil foto truk menggunakan kamera dalam aplikasi. Sistem menjalankan OCR (Google ML Kit) secara on-device untuk membaca pelat nomor secara otomatis. Hasil OCR ditampilkan ke petugas untuk diverifikasi atau dikoreksi. Foto kedua untuk muatan tebu juga diambil sebagai dokumentasi visual. Metadata GPS dan timestamp dari foto ikut tersimpan dan divalidasi per entri truk.

Langkah 5-6: Submit & Server Validation DIPERBARUI

Setelah petugas menekan tombol Submit, seluruh data dikirim ke backend API beserta metadata lokasi GPS, device timestamp, dan device ID. Server melakukan validasi keamanan berlapis:

1) Validasi pos binding — cek bahwa petugas_id yang submit memiliki pos_id yang sama dengan pos_id dalam data transaksi. Jika tidak cocok, transaksi ditolak.
2) Server timestamp — waktu pencatatan menggunakan NOW() dari server, bukan timestamp dari device, untuk mencegah manipulasi waktu.
3) Cek duplikasi (apakah truk yang sama sudah pernah tercatat hari ini di pos yang sama).
4) Cek konsistensi kontrak serta menyimpan foto ke cloud storage.
5) Foto dan GPS metadata divalidasi dan disimpan per entri — setiap record memiliki bukti lokasi dan visual yang lengkap.

Langkah 7: Notifikasi WhatsApp — Ringkasan Harian Jam 22:00 WIB DIPERBARUI

Notifikasi WhatsApp tidak lagi dikirim per truk secara real-time. Sebagai gantinya, sistem menggunakan cron job yang berjalan setiap hari jam 22:00 WIB untuk mengumpulkan seluruh data truk yang masuk hari itu, lalu mengirimkan satu pesan ringkasan per nomor kontrak. Ini mencegah spam jika satu kontrak mengirim banyak truk dalam satu hari. Pesan berisi total truk, daftar pelat nomor, dan pos-pos yang dilewati.

Alur Khusus — Tebu Lokal (Asal Kudus)

Truk tebu lokal dari wilayah Kudus tidak melewati pos pantau dan langsung menuju area timbangan/penerimaan di pabrik. Data tebu lokal dicatat langsung oleh petugas pabrik melalui sistem timbangan internal, sehingga tidak termasuk dalam cakupan aplikasi pos pantau ini. Namun, kedua alur (lokal & luar) akan bertemu di database pusat untuk pelaporan terpadu.

A. Validasi Backend — Pos Binding

Middleware: Validate Pos Assignment

Setiap request ke endpoint POST /api/transaksi melewati middleware yang memastikan petugas hanya bisa submit ke pos yang ditugaskan.

// Middleware: validatePosAssignment async function validatePosAssignment(req, res, next) { // 1. Ambil petugas_id dari JWT token (bukan dari request body) const petugasId = req.user.id; // 2. Query pos_id yang ditugaskan ke petugas ini const petugas = await db.query( 'SELECT pos_id FROM petugas WHERE id = $1 AND is_active = TRUE', [petugasId] ); if (!petugas.rows[0]) { return res.status(403).json({ error: 'Akun petugas tidak aktif atau tidak ditemukan' }); } // 3. Bandingkan pos_id petugas dengan pos_id di request const assignedPosId = petugas.rows[0].pos_id; const requestedPosId = req.body.pos_id; if (assignedPosId !== requestedPosId) { // LOG: Percobaan submit lintas pos (possible fraud) await logSecurityEvent({ type: 'CROSS_POS_ATTEMPT', petugas_id: petugasId, assigned_pos: assignedPosId, attempted_pos: requestedPosId, ip: req.ip, timestamp: new Date() }); return res.status(403).json({ error: 'Anda tidak ditugaskan di pos ini', detail: 'Hubungi admin jika ada perubahan penugasan' }); } // 4. Inject data terverifikasi ke request req.verifiedPosId = assignedPosId; req.verifiedPetugasId = petugasId; next(); }

B. Kebijakan Timestamp

Dual Timestamp Strategy

Sistem menyimpan dua timestamp untuk setiap transaksi:

FieldSumberTujuanBisa Dimanipulasi?
server_timestamp NOW() di PostgreSQL Timestamp resmi — digunakan untuk laporan, audit, dan notifikasi TIDAK
device_timestamp Clock di HP petugas Referensi — untuk deteksi anomali jika ada selisih besar dengan server YA (oleh petugas)

Jika selisih antara device_timestamp dan server_timestamp lebih dari 10 menit, sistem otomatis membuat flag time_anomaly = TRUE untuk ditinjau admin.

Solusi Teknis

Detail implementasi teknis untuk fitur keamanan anti-fake GPS, OCR pelat nomor, dan integrasi notifikasi WhatsApp.

A. Anti-Fake GPS — Strategi Multi-Layer

Manipulasi GPS (fake GPS) adalah risiko utama di lapangan. Solusinya bukan hanya satu metode, tetapi kombinasi beberapa layer pertahanan:

Layer 1 — Mock Location Provider Detection

Pada Android, fake GPS bekerja sebagai "Mock Location Provider". Aplikasi mendeteksi ini melalui:

// Android — Cek apakah Mock Location aktif fun isMockLocationEnabled(location: Location): Boolean { // API 18+ — cek flag isFromMockProvider if (Build.VERSION.SDK_INT >= 18) { return location.isFromMockProvider } // Fallback — cek Settings return Settings.Secure.getString( context.contentResolver, Settings.Secure.ALLOW_MOCK_LOCATION ) != "0" }

Jika terdeteksi mock location, aplikasi langsung menolak dan menampilkan peringatan.

Layer 2 — Sensor Fusion Verification

Fake GPS hanya memalsukan koordinat, tetapi tidak bisa memalsukan data sensor perangkat. Sistem melakukan cross-check:

SensorPengecekanIndikasi Fake
Cell Tower IDBandingkan lokasi GPS vs lokasi cell tower terdekatJarak GPS vs cell tower > 5 km
Wi-Fi SSIDScan Wi-Fi sekitar, cocokkan dengan database posWi-Fi pos tidak terdeteksi
AccelerometerDeteksi gerakan natural (HP dipegang orang berdiri)Sensor statis sempurna (emulator)
NMEA SatelliteCek jumlah satelit GPS yang terhubungJumlah satelit = 0 (mock)

Layer 3 — Geofence Validation (Server-Side)

Server menyimpan koordinat presisi setiap pos pantau. Saat data masuk, server menghitung jarak antara koordinat GPS yang dilaporkan dengan koordinat pos pantau menggunakan formula Haversine:

// Server-side geofence validation function validateGeofence(reportedLat, reportedLng, postId) { const post = getPostCoordinates(postId); const distance = haversineDistance( reportedLat, reportedLng, post.lat, post.lng ); const MAX_RADIUS = 200; // meter if (distance > MAX_RADIUS) { throw new Error(`Lokasi diluar radius pos (${distance}m)`); } return { valid: true, distance }; }

Radius toleransi: 200 meter dari titik pusat pos pantau. Jika diluar radius, transaksi ditolak.

Layer 4 — Foto Metadata & Integrity Check

Foto yang diambil harus memiliki EXIF metadata dengan GPS coordinates, timestamp, dan device info. Sistem memastikan foto diambil langsung dari kamera aplikasi (bukan dari galeri) dengan meng-embed watermark invisible berisi hash koordinat GPS + timestamp. Ini mencegah petugas mengirim foto lama atau foto dari lokasi lain.

Layer 5 — Anomaly Detection (Opsional Lanjutan)

Sistem backend memantau pola abnormal, seperti: petugas yang selalu submit tepat di koordinat yang sama persis (natural GPS selalu bervariasi 5-15 meter), waktu perjalanan antar submit yang tidak masuk akal (misalnya submit di Pos Jepara dan 10 menit kemudian di Pos Pati), atau lonjakan submit di luar jam operasional.

B. OCR Pelat Nomor — Identifikasi Truk Otomatis

Arsitektur OCR

Menggunakan pendekatan hybrid on-device + server:

KomponenTeknologiKeterangan
On-Device (Primary)Google ML Kit Text RecognitionBerjalan di HP tanpa internet, latensi rendah (~300ms)
Server FallbackGoogle Cloud Vision API / PaddleOCRDipanggil jika on-device gagal atau confidence < 80%
Post-ProcessingRegex Pattern MatchingValidasi format pelat Indonesia: AB 1234 CD

Alur OCR

1) Petugas mengarahkan kamera ke pelat nomor → 2) ML Kit mendeteksi area teks secara real-time → 3) Hasil OCR difilter dengan regex pelat Indonesia /^[A-Z]{1,2}\s?\d{1,4}\s?[A-Z]{0,3}$/ → 4) Jika confidence ≥ 80%, nomor otomatis terisi ke field input → 5) Petugas memverifikasi dan koreksi jika perlu → 6) Jika confidence < 80%, foto dikirim ke server untuk diproses ulang.

C. Integrasi WhatsApp — Ringkasan Harian Jam 22:00 WIB DIPERBARUI

Perubahan Mekanisme Pengiriman

Sebelumnya: pesan WhatsApp dikirim per truk secara real-time setiap kali data di-submit. Masalah: jika satu kontrak mengirim 10 truk dalam sehari, petani menerima 10 pesan terpisah — ini mengganggu dan terkesan spam.

Mekanisme baru: Semua data truk dikumpulkan sepanjang hari, lalu satu pesan ringkasan dikirim per nomor kontrak pada jam 22:00 WIB. Petani hanya menerima maksimal 1 pesan per hari per kontrak.

Cron Job — Kirim Ringkasan Jam 22:00 WIB BARU

// Cron Job: Setiap hari jam 22:00 WIB (UTC+7) // Crontab: 0 22 * * * (atau 0 15 * * * UTC) async function sendDailyWhatsAppSummary() { const today = new Date().toISOString().slice(0, 10); // 1. Ambil semua transaksi hari ini, grup per kontrak const summaries = await db.query(` SELECT k.no_kontrak, p.nama AS nama_petani, p.no_wa, COUNT(t.id) AS jumlah_truk, JSON_AGG( JSON_BUILD_OBJECT( 'no_plat', t.no_plat, 'nama_pos', pp.nama_pos, 'waktu', TO_CHAR(t.server_timestamp, 'HH24:MI') ) ORDER BY t.server_timestamp ) AS daftar_truk FROM transaksi_pos t JOIN kontrak k ON t.kontrak_id = k.id JOIN petani p ON k.petani_id = p.id JOIN pos_pantau pp ON t.pos_id = pp.id WHERE DATE(t.server_timestamp) = $1 AND t.wa_status = 'pending' GROUP BY k.no_kontrak, p.nama, p.no_wa `, [today]); // 2. Kirim 1 pesan per kontrak for (const summary of summaries.rows) { const message = buildDailySummaryTemplate(summary); try { await fetch('https://api.fonnte.com/send', { method: 'POST', headers: { 'Authorization': process.env.FONNTE_TOKEN, 'Content-Type': 'application/json' }, body: JSON.stringify({ target: summary.no_wa, message: message, countryCode: '62' }) }); // Update semua transaksi kontrak ini hari ini await db.query(` UPDATE transaksi_pos SET wa_status = 'sent', wa_sent_at = NOW() WHERE kontrak_id = ( SELECT id FROM kontrak WHERE no_kontrak = $1 ) AND DATE(server_timestamp) = $2 `, [summary.no_kontrak, today]); } catch (err) { // Masukkan ke retry queue (max 3 retry) await addToRetryQueue(summary.no_kontrak, today, 3); } } console.log(`[${today}] Ringkasan WA terkirim: ${summaries.rows.length} kontrak`); }

Sistem retry: jika pengiriman gagal, pesan masuk ke queue dan dicoba ulang hingga 3 kali dengan interval 5 menit. Status pengiriman disimpan di database untuk audit trail. Jika semua retry gagal, status berubah ke failed dan admin mendapat notifikasi.

-- ============================================= -- TABEL: transaksi_pos (UPDATED — tambah server/device timestamp) -- ============================================= CREATE TABLE transaksi_pos ( id SERIAL PRIMARY KEY, kontrak_id INT REFERENCES kontrak(id), pos_id INT REFERENCES pos_pantau(id) NOT NULL, petugas_id INT REFERENCES petugas(id) NOT NULL, no_plat VARCHAR(15) NOT NULL, foto_truk_url TEXT, foto_muatan_url TEXT, gps_lat DECIMAL(10,8), gps_lng DECIMAL(11,8), gps_accuracy DECIMAL(6,2), is_mock_detected BOOLEAN DEFAULT FALSE, device_id VARCHAR(100), server_timestamp TIMESTAMP DEFAULT NOW(), -- BARU: waktu resmi dari server device_timestamp TIMESTAMP, -- BARU: waktu dari HP (referensi) time_anomaly BOOLEAN DEFAULT FALSE, -- BARU: flag jika selisih > 10min wa_status VARCHAR(10) DEFAULT 'pending', wa_sent_at TIMESTAMP, catatan TEXT, created_at TIMESTAMP DEFAULT NOW(), -- CONSTRAINT: pastikan pos_id petugas cocok (enforced via trigger) CONSTRAINT chk_pos_petugas CHECK (pos_id IS NOT NULL AND petugas_id IS NOT NULL) ); -- TRIGGER: Validasi bahwa petugas.pos_id == transaksi.pos_id CREATE OR REPLACE FUNCTION validate_pos_binding() RETURNS TRIGGER AS $$ DECLARE petugas_pos INT; BEGIN SELECT pos_id INTO petugas_pos FROM petugas WHERE id = NEW.petugas_id AND is_active = TRUE; IF petugas_pos IS NULL THEN RAISE EXCEPTION 'Petugas tidak aktif atau tidak ditemukan'; END IF; IF petugas_pos != NEW.pos_id THEN RAISE EXCEPTION 'Petugas (%) tidak ditugaskan di pos (%). Ditugaskan di pos (%)', NEW.petugas_id, NEW.pos_id, petugas_pos; END IF; -- Auto-detect time anomaly IF NEW.device_timestamp IS NOT NULL AND ABS(EXTRACT(EPOCH FROM (NOW() - NEW.device_timestamp))) > 600 THEN NEW.time_anomaly := TRUE; END IF; RETURN NEW; END; $$ LANGUAGE plpgsql; CREATE TRIGGER trg_validate_pos_binding BEFORE INSERT ON transaksi_pos FOR EACH ROW EXECUTE FUNCTION validate_pos_binding(); -- ============================================= -- TABEL: security_log (BARU) -- ============================================= CREATE TABLE security_log ( id SERIAL PRIMARY KEY, event_type VARCHAR(50) NOT NULL, petugas_id INT REFERENCES petugas(id), pos_id INT REFERENCES pos_pantau(id), detail JSONB, ip_address INET, created_at TIMESTAMP DEFAULT NOW() ); -- Index untuk performa query CREATE INDEX idx_transaksi_kontrak ON transaksi_pos(kontrak_id); CREATE INDEX idx_transaksi_pos ON transaksi_pos(pos_id); CREATE INDEX idx_transaksi_tanggal ON transaksi_pos(server_timestamp); CREATE INDEX idx_transaksi_wa_status ON transaksi_pos(wa_status, server_timestamp); CREATE INDEX idx_kontrak_no ON kontrak(no_kontrak); CREATE INDEX idx_security_log_type ON security_log(event_type, created_at); CREATE INDEX idx_security_log_petugas ON security_log(petugas_id, created_at);

Query — Dashboard Petugas Pos BARU

Query: Data Truk Hari Ini di Pos Petugas

-- Query untuk dashboard petugas: data truk hari ini di posnya SELECT t.id, t.no_plat, t.server_timestamp AS waktu_masuk, k.no_kontrak, p.nama AS nama_petani, pt.nama AS diinput_oleh, t.wa_status, t.gps_accuracy, t.time_anomaly FROM transaksi_pos t LEFT JOIN kontrak k ON t.kontrak_id = k.id LEFT JOIN petani p ON k.petani_id = p.id JOIN petugas pt ON t.petugas_id = pt.id WHERE t.pos_id = :user_pos_id -- hanya pos petugas yg login AND DATE(t.server_timestamp) = :tanggal_filter ORDER BY t.server_timestamp DESC; -- Ringkasan harian SELECT COUNT(*) AS total_truk_hari_ini, COUNT(DISTINCT t.no_plat) AS truk_unik, COUNT(CASE WHEN t.wa_status = 'sent' THEN 1 END) AS wa_terkirim, COUNT(CASE WHEN t.wa_status = 'failed' THEN 1 END) AS wa_gagal FROM transaksi_pos t WHERE t.pos_id = :user_pos_id AND DATE(t.server_timestamp) = CURRENT_DATE;

Catatan Dashboard Web BARU

Dashboard web diakses melalui browser di HP atau laptop petugas. Fitur utama: (1) ringkasan harian di bagian atas (total truk, truk unik, status WA), (2) tabel data truk dengan kolom Nomor Polisi, Waktu Masuk, No. Kontrak, Nama Petani, Siapa yang Input, dan Status WA, (3) filter tanggal untuk melihat data hari sebelumnya, (4) search bar untuk pencarian cepat. Data di-scope ke pos_id petugas yang login — petugas tidak bisa melihat data pos lain.
